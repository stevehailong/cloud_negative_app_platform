# 云原生应用研发交付平台 — 技术难点分析与优化建议

> 基于代码审查、架构分析和实际运行验证的全面评估

---

## 一、技术难点总结

### 1.1 网络连通性：Docker ↔ K8s 混合部署的天然鸿沟

**问题描述**：
平台服务运行在 Docker Compose 中，用户应用部署在 Docker Desktop 内置 K8s 中，两者网络隔离。

| 通信方向 | 问题 | 当前方案 |
|----------|------|----------|
| K8s Pod → MySQL | Pod 内 `127.0.0.1` 不可达 | socat sidecar 转发 |
| K8s Pod → Registry | containerd 不支持 HTTP | DaemonSet 注入 hosts.toml |
| Docker → K8s Service | ClusterIP 不可达 | kubectl port-forward |
| K8s Pod → Docker 服务 | DNS 不互通 | 硬编码 172.18.0.1 |
| Prometheus → K8s cAdvisor | API 不可达 | kubectl proxy 守护进程 |

**技术债**：5 个方向各用不同方案，维护成本高，重启后易失效。

**优化建议**：
```yaml
# 1. 将所有平台服务迁移到 K8s，统一网络平面
# 2. MySQL/Redis 创建 K8s Service + Endpoints 指向外部实例
# 3. Registry 直接部署在 K8s 中，使用 ClusterIP 访问
# 4. 使用 K8s DNS (service.namespace.svc.cluster.local) 替代硬编码 IP
```

### 1.2 数据库：15 个独立 DB 的事务与跨库查询

**问题描述**：
设计文档要求每服务一库（15 个 DB），但实际存在大量跨库查询：

```sql
-- 实际出现的跨库查询模式（cluster-service handler）
SELECT app_id FROM deploy_db.app_deployments WHERE namespace=?
SELECT project_id FROM app_db.applications WHERE id=?
-- 审计中间件写 audit_db，但用户信息在 iam_db
-- 成本数据需关联 project/application 但分布在 app_db/org_db
```

**痛点**：
- 无法使用事务保证一致性
- 跨库 JOIN 只能应用层拼接
- 15 个 DB 连接池管理混乱
- 数据同步延迟不可控

**优化建议**：
```
方案A（推荐）: 合并为 3-4 个核心 DB
  ├── platform_db   (iam, org, app, env, infra)
  ├── cicd_db       (devops, release, deploy)
  ├── observability_db (monitor, audit, notification)
  └── governance_db (resource, config, secret, cost)

方案B: 保留分库，引入统一数据服务层
  - 所有跨库查询通过 GraphQL/API Gateway 聚合
  - 数据同步使用 Debezium CDC + Kafka
```

### 1.3 前端响应拦截器与后端 API 的不一致

**问题描述**：
axios 响应拦截器（`request.js:48`）返回 `response.data`（解包后为 `{code, message, data}`），但多个页面仍用 `res.data.code` 等冗余路径，导致数据显示不出来。

本次修复涉及的文件：
- `ClusterList.vue` — `res.data.data.list` → `res.data.list`
- `ClusterMonitor.vue` — 硬编码 mock 数据 → 真实 API
- `AuditLogs.vue` / `MonitorDashboard.vue` — 格式正确无需修复

**根因**：缺少统一的 API 层封装，拦截器行为对开发者不透明。

**优化建议**：
```typescript
// 统一封装 API 调用，彻底消除 res.data.data 问题
// api.ts
export async function apiGet<T>(url: string, params?: any): Promise<PageResult<T>> {
  const res = await request.get(url, { params })
  // request 拦截器已解包为 {code, message, data}
  if (res.code !== 0 && res.code !== 200) throw new ApiError(res.code, res.message)
  return res.data  // 直接返回业务 data
}

// 组件中直接使用
const clusters = await apiGet('/clusters', { page: 1, pageSize: 10 })
tableData.value = clusters.list
pagination.total = clusters.total
```

### 1.4 镜像构建：硬编码的 127.0.0.1 与跨环境不可移植

**问题描述**：
`stevehailong-project.git` 构建的 `book_server` 二进制硬编码：
```
root:root123456@tcp(127.0.0.1:3306)/books
```
导致该镜像在 Docker Compose 正常，部署到 K8s 就 CrashLoopBackOff。

**根因**：Jenkins 流水线没有注入环境变量覆盖机制。

**优化建议**：
```go
// 1. 要求所有应用读取 DB_DSN 环境变量，不硬编码连接字符串
// 在 Jenkins pipeline 中自动注入
dbDSN := os.Getenv("DB_DSN")
if dbDSN == "" {
    dbDSN = "root:root123456@tcp(127.0.0.1:3306)/books?charset=utf8mb4&parseTime=True"
}

// 2. 构建时注入默认值
// Dockerfile ARG + ENV 模式
ARG DB_HOST=127.0.0.1
ENV DB_HOST=$DB_HOST
```

### 1.5 可观测性：多套系统的碎片化

**问题描述**：

| 组件 | 当前方案 | 问题 |
|------|----------|------|
| 指标 | Prometheus(9091) | kubectl proxy 不稳定 |
| 日志 | Nginx access_trace.log | trace-log-reader 二进制不工作，自写 shell 替代 |
| 链路 | 自建 trace_spans 表 | 格式不标准，Nginx JSON → TraceSpan 需 awk 转换 |
| 监控 | Grafana(3000) | 无预配置 Dashboard |
| 审计 | Gateway 中间件写 DB | 直接耦合 DB，非生产级 |

**优化建议**：
```
日志：采用 Loki + Promtail 替代自建方案
  - Docker Compose 服务 → Promtail → Loki
  - K8s Pod → Promtail DaemonSet → Loki
  - Grafana 统一查询面板

链路：采用 OpenTelemetry + Jaeger 替代自建 trace_spans
  - Gin 中间件自动注入 TraceID
  - Gateway 生成根 Span，下游服务创建子 Span
  - 标准化 W3C Trace Context 传播

审计：采用异步消息队列
  - Gateway 中间件 → Kafka topic → audit-service consumer → audit_db
```

### 1.6 Jenkins 与 Docker 的权限和网络

**问题描述**：
- Jenkins 通过挂载 `/var/run/docker.sock` 使用宿主机 Docker
- `docker push host.docker.internal:5001` 在 Mac Docker Desktop 上不稳定
- Jenkins Job XML 硬编码在 Go 代码中（120 行模板字符串）

**优化建议**：
```go
// 1. 使用 Jenkinsfile (Pipeline as Code) 替代硬编码 Job XML
// 2. 支持 GitLab CI / GitHub Actions 作为 CI 引擎
// 3. 构建任务容器化（Kaniko/Buildpacks），不再依赖 Docker socket
// 4. 制品推送地址通过环境变量注入
```

## 二、代码质量优化

### 2.1 硬编码问题

| 位置 | 硬编码 | 影响 |
|------|--------|------|
| `pipeline_service.go:343` | `172.18.0.1:5001` | 镜像地址不可移植 |
| `pipeline_service.go:414` | `host.docker.internal:5001` | Docker Desktop 特供 |
| `cluster-service/main.go` | `root:root123456@tcp(mysql:3306)` | 数据库密码硬编码 |
| `gateway/main.go` | 同样 | 15 个服务全硬编码 |
| `pipeline_service.go:120` | Jenkins Job XML 模板 | 120 行字符串 |

**优化建议**：
```bash
# 统一配置管理
export REGISTRY_URL=172.18.0.1:5001     # 环境变量
export MYSQL_PASSWORD=root123456
# 或使用 Nacos / config.yaml 文件
```

### 2.2 `%1000` vs `%10000` 的时间戳 Bug

`pipeline_service.go:515/527` 使用 `time.Now().Unix()%1000` 而 `:266` 使用 `%10000`，导致 `artifact_version` 与 `image_tag` 不匹配。已修复但揭示问题是缺少共享的版本生成函数。

**优化建议**：
```go
// 抽取统一的版本生成函数
func GenerateVersion(timestamp int64, commit string) string {
    return fmt.Sprintf("1.0.%d-%s", timestamp%10000, commit[:7])
}
```

### 2.3 数据库 JSON 列空值处理

MySQL 8.0 严格模式拒绝 JSON 列的空字符串 `''`：
- `trace_spans.tags` → `Tags *string`（已修复）
- `namespaces.resource_quota_json` → 需设置 `"{}"`（已修复）

**优化建议**：统一 Model 规范，JSON 列使用 `*string` 类型。

## 三、安全优化

### 3.1 数据库密码硬编码

15 个 main.go 文件中全部硬编码 `root:root123456`。

**优化建议**：
```go
// 统一从环境变量读取
dsn := fmt.Sprintf("%s:%s@tcp(%s:%s)/%s?charset=utf8mb4&parseTime=True",
    os.Getenv("DB_USER"), os.Getenv("DB_PASS"), os.Getenv("DB_HOST"), os.Getenv("DB_PORT"), dbName)
```

### 3.2 JWT Secret 硬编码

`my-cloud-secret-key-change-this-in-production` 在多处重复。

**优化建议**：使用 Vault 或 Kubernetes Secret 注入。

### 3.3 Registry 无认证

本地 Registry 完全开放，无 TLS，无访问控制。

**优化建议**：升级为 Harbor，或至少配置 HTTP basic auth + TLS。

## 四、稳定性优化

### 4.1 守护进程管理

当前有多个必须手动维护的后台进程：

| 进程 | 作用 | 风险 |
|------|------|------|
| `kubectl proxy :8001` | Prometheus 访问 K8s | 终端关闭即死 |
| `kubectl port-forward :8888` | Ingress 对外暴露 | 终端关闭即死 |
| `trace-forwarder.sh` | Nginx Trace 上报 | 容器重启需恢复 |

**优化建议**：
```yaml
# docker-compose 中增加 sidecar 容器管理这些进程
# 或使用 systemd/launchd 管理宿主机守护进程
# 或全部迁移到 K8s 中（以上进程不需要了）
```

### 4.2 启动顺序依赖

15 个服务全部依赖 MySQL + Redis 先启动，但 `depends_on: condition: service_healthy` 只保证启动顺序，不保证初始化完成。

**优化建议**：每个服务增加数据库连接重试机制（带退避策略）。

### 4.3 Docker Desktop 重启后状态丢失

K8s 节点容器重建后：
- containerd certs.d 配置丢失（已通过 DaemonSet 修复）
- kubectl proxy 端口可能变化
- Ingress port-forward 需要重建

**优化建议**：创建 Docker Desktop 启动后自动运行的 bootstrap 脚本。
