# 一、平台启动与访问

## 1.1 环境要求

- Docker 20.10+
- Docker Compose 2.0+
- 可用内存 4GB+
- 端口：80（前端）、8080（API网关）、3306（MySQL）、6379（Redis）

## 1.2 启动平台

```bash
# 进入项目目录
cd /path/to/my_cloud

# 一键启动所有服务
docker-compose up -d

# 查看服务运行状态
docker-compose ps
```

首次启动需等待：
- MySQL 数据库初始化（约30秒）
- Docker 镜像构建（约3-5分钟）
- 各微服务启动（约10秒）

## 1.3 验证服务就绪

```bash
# 检查网关
curl http://localhost:8080/health

# 检查各服务健康状态
curl http://localhost:8081/health  # Auth Service
curl http://localhost:8082/health  # Project Service
curl http://localhost:8083/health  # Application Service
curl http://localhost:8084/health  # Pipeline Service
curl http://localhost:8085/health  # Env Service
curl http://localhost:8086/health  # Release Service
curl http://localhost:8087/health  # Deploy Service
curl http://localhost:8088/health  # Cluster Service
```

## 1.4 访问平台

- 打开浏览器访问：**http://localhost**
- 使用默认账号登录：`admin` / `admin123`

## 1.5 停止与清理

```bash
# 停止所有服务
docker-compose down

# 停止并清除所有数据（慎用）
docker-compose down -v
```

# 二、账号与登录

## 2.1 登录

1. 打开平台首页 http://localhost，进入登录页面
2. 输入用户名和密码
3. 点击"登录"按钮
4. 登录成功后自动跳转到工作台

**默认管理员账号：** admin / admin123

## 2.2 注册新用户

1. 在登录页面点击"注册"链接，跳转到注册页面
2. 填写以下信息：
   - 用户名（必填，唯一）
   - 密码（必填）
   - 邮箱（必填）
   - 真实姓名（选填）
3. 点击"注册"完成账号创建
4. 注册成功后可直接登录

## 2.3 Token 机制

- 登录成功后系统发放 `access_token`（24小时有效）和 `refresh_token`（7天有效）
- Token 过期时系统会自动尝试刷新，无需手动操作
- 如果 refresh_token 也过期，需要重新登录

## 2.4 退出登录

- 点击页面右上角的用户头像
- 在下拉菜单中选择"退出登录"

# 四、组织与项目管理

## 4.1 租户管理

**导航路径：** 侧边栏 > 组织与项目 > 租户管理

租户是平台的顶级资源隔离单位，一般对应一个企业或独立团队。

### 操作指南

| 操作 | 步骤 |
|------|------|
| 查看租户列表 | 进入租户管理页面，可分页浏览所有租户 |
| 创建租户 | 点击"新建租户"按钮，填写租户编码、名称、描述，选择拥有者 |
| 编辑租户 | 在列表中点击对应租户的"编辑"按钮 |
| 删除租户 | 在列表中点击"删除"按钮（需确认） |

### 字段说明

- **租户编码**：唯一标识，如 `acme`，创建后不可修改
- **租户名称**：显示名称，如 "Acme集团"
- **拥有者**：该租户的管理员用户
- **描述**：租户用途说明

## 4.2 项目管理

**导航路径：** 侧边栏 > 组织与项目 > 项目管理

项目是应用和团队成员组织的核心单元，一个租户下可以有多个项目。

### 操作指南

| 操作 | 步骤 |
|------|------|
| 查看项目列表 | 进入项目管理页面，支持搜索和分页 |
| 创建项目 | 点击"新建项目"，填写项目编码、名称、所属租户、负责人、描述 |
| 编辑项目 | 点击列表中的"编辑"按钮 |
| 删除项目 | 点击"删除"按钮 |
| 管理项目成员 | 进入项目详情，在"成员"标签页添加/移除成员 |

### 字段说明

- **项目编码**：唯一标识，如 `paas-platform`
- **项目名称**：显示名称
- **所属租户**：项目归属的租户
- **负责人**：项目 Owner

### 项目成员管理

1. 进入项目详情页
2. 切换到"成员"标签
3. 点击"添加成员"，选择用户并分配角色
4. 支持移除已有成员

# 五、应用管理

**导航路径：** 侧边栏 > 应用管理

应用是平台的核心概念，代表一个可独立部署的微服务或业务系统。

## 5.1 应用列表

进入应用管理页面，可查看所有应用，支持按名称/编码搜索、分页浏览。

## 5.2 创建应用

1. 点击"新建应用"按钮
2. 填写应用信息：
   - **应用名称**：如 "用户中心"
   - **应用编码**：唯一标识，如 `user-center`
   - **所属项目**：选择关联的项目
   - **应用类型**：web / api / worker / job
   - **开发语言**：go / java / python / nodejs 等
   - **代码仓库地址**：GitLab 仓库 URL
   - **默认分支**：如 `main`
   - **描述**：应用功能说明
3. 点击"确认"创建应用

## 5.3 应用详情

点击应用名称进入应用详情页，包含以下信息：
- **基本信息**：应用编码、类型、语言、仓库地址等
- **组件列表**：应用的部署组件（如 API Server、Worker 等）
- **环境绑定**：应用在各环境中的部署配置
- **依赖关系**：应用间的依赖拓扑

## 5.4 管理应用组件

应用组件定义了应用在 Kubernetes 中的部署形态。

1. 在应用详情页切换到"组件"标签
2. 点击"添加组件"：
   - **组件编码**：如 `api-server`
   - **组件类型**：deployment / statefulset / job
   - **镜像名称**：容器镜像地址
   - **副本数**：期望实例数
   - **资源配额**：CPU/内存 的 request 和 limit
   - **端口配置**：容器暴露的端口
   - **环境变量**：容器运行时的环境变量

## 5.5 管理应用依赖

1. 在应用详情页切换到"依赖"标签
2. 点击"添加依赖"，选择被依赖的应用
3. 依赖关系可用于生成应用拓扑图

# 六、流水线（CI/CD）

**导航路径：** 侧边栏 > 流水线

流水线实现代码从提交到制品构建的自动化过程。

## 6.1 查看流水线

进入流水线页面，查看所有已创建的流水线，包括名称、类型、关联应用、最近执行状态等。

## 6.2 创建流水线

1. 点击页面右上角的"新建流水线"按钮
2. 在弹出的对话框中填写配置：
   - **流水线名称**（必填）：如 "用户中心-CI"
   - **流水线编码**（必填）：唯一标识，如 `user-center-ci`
   - **关联应用ID**（必填）：关联的应用 ID
   - **流水线类型**：ci（构建）/ cd（部署）/ ci_cd（完整）
   - **CI 工具**：Jenkins / Tekton
   - **Git仓库URL**：代码仓库地址
   - **默认分支**：如 `main`
   - **状态**：启用/禁用开关
3. 点击"确认"保存流水线

## 6.3 执行流水线

1. 在流水线列表中找到目标流水线
2. 点击"运行"按钮触发构建
3. 流水线执行后可查看：
   - **运行记录**：所有历史执行列表
   - **执行日志**：单次运行的详细日志
   - **产出制品**：构建生成的镜像/包

## 6.4 查看制品

流水线构建成功后会产出制品（Docker 镜像），制品信息包括：
- 制品版本
- 镜像地址
- 构建时间
- 关联的流水线运行 ID

制品可在后续发布流程中选择使用。

# 八、部署管理

**导航路径：** 侧边栏 > 部署管理

部署管理展示应用在各环境中的实际运行状态。

## 8.1 查看部署列表

展示所有部署记录，包括：
- 应用名称
- 目标环境
- 镜像版本
- 部署状态（progressing / success / failed / rollback）
- 副本数（期望 / 实际）
- 部署时间

## 8.2 部署操作

| 操作 | 说明 |
|------|------|
| 重启 | 对部署执行滚动重启 |
| 扩缩容 | 修改副本数，水平伸缩 |
| 查看事件 | 查看 K8s 部署事件 |
| 查看 Pod | 查看属于该部署的所有 Pod 实例 |
| 查看日志 | 查看 Pod 实时日志 |
| 回滚 | 回退到上一个版本 |

## 8.3 发布流程（完整的应用发布）

**导航路径：** 侧边栏 > 发布管理

平台推荐的标准发布流程如下：

```
代码推送 → CI构建 → 部署上线(创建发布单) → 提交审批 → 审批通过 → 执行发布 → 金丝雀部署 → 确认全量 → 发布完成
```

### 步骤详解

1. **CI 构建产出制品**：流水线执行成功后产出 Docker 镜像制品
2. **创建发布单**：在流水线列表中点击"部署上线"按钮，系统自动取最新制品创建发布工单
3. **提交审批**：进入发布管理页面，点击"提交审批"按钮
4. **审批通过**：审批人在发布管理页面点击"审批通过"
5. **执行发布**：审批通过后点击"执行发布"，系统自动执行金丝雀部署：
   - 创建带 `-canary` 后缀的独立工作负载（1个副本）
   - 承接约 20% 的线上流量
   - 系统自动等待部署就绪
6. **确认全量/回滚**：
   - **确认全量**：验证金丝雀无异常后，点击"确认全量"，系统自动：
     - 部署正式工作负载（全量副本数）
     - 清理金丝雀工作负载
   - **回滚金丝雀**：发现问题时点击"回滚金丝雀"，系统自动缩容 canary 副本到 0

### 发布状态说明

| 状态 | 含义 | 可执行操作 |
|------|------|-----------|
| created | 已创建，待提交 | 提交审批 |
| submitted | 已提交，待审批 | 审批通过 / 拒绝 |
| approved | 已通过，待执行 | 执行发布 |
| executing | 执行中 | 等待 |
| canary | 金丝雀运行中 | 确认全量 / 回滚 |
| success | 发布成功 | 回滚 |
| rollback | 已回滚 | - |
| failed | 发布失败 | 重新创建 |

### 发布策略

| 策略 | 说明 |
|------|------|
| canary | 金丝雀发布：先部署少量实例验证，确认后全量发布（推荐） |
| rolling | 滚动更新：逐批替换旧版本实例 |
| blue-green | 蓝绿部署：新旧版本并行，切换流量后下线旧版本 |

# 十三、常见问题与故障排查

## 13.1 无法访问平台

**问题**：浏览器打开 http://localhost 显示无法连接

**排查步骤**：
```bash
# 检查容器是否运行
docker-compose ps

# 检查前端容器日志
docker logs my-cloud-frontend

# 检查端口是否被占用
lsof -i :80
```

## 13.2 登录失败

**问题**：输入账号密码后提示认证失败

**排查步骤**：
```bash
# 检查 Auth Service 是否正常
curl http://localhost:8081/health

# 检查 Gateway 是否正常
curl http://localhost:8080/health

# 查看 Auth Service 日志
docker logs my-cloud-auth-service --tail 20
```

## 13.3 API 返回 502

**问题**：前端操作时提示网关错误

**排查步骤**：
```bash
# 查看 Gateway 日志
docker logs my-cloud-gateway --tail 50

# 检查对应的后端服务是否存活
docker-compose ps

# 重启出问题的服务
docker-compose restart <service-name>
```

## 13.4 数据库连接失败

```bash
# 检查 MySQL 是否正常
docker exec -it my-cloud-mysql mysqladmin ping -uroot -proot123456

# 进入 MySQL 查看数据库
docker exec -it my-cloud-mysql mysql -uroot -proot123456 -e "SHOW DATABASES;"

# 查看 MySQL 日志
docker logs my-cloud-mysql --tail 20
```

## 13.5 服务重建

如果某个服务异常，可以重新构建和启动：
```bash
# 重新构建镜像并启动
docker-compose up -d --build <service-name>

# 例如重建网关
docker-compose up -d --build gateway
```

## 13.6 完全重置

如果需要完全从零开始：
```bash
# 停止所有容器并删除数据卷
docker-compose down -v

# 重新启动（会重新初始化数据库）
docker-compose up -d
```

# 附录：典型使用场景操作流程

## 场景一：新项目从零开始部署上线

```
1. 创建租户（如已有可跳过）
2. 创建项目，添加项目成员
3. 创建应用，填写代码仓库地址
4. 添加应用组件（定义部署规格）
5. 创建环境（dev/staging/prod）
6. 接入 K8s 集群（如已有可跳过）
7. 创建流水线，关联应用
8. 执行流水线，构建制品
9. 创建发布单，选择制品和目标环境
10. 提交审批 → 审批通过 → 执行发布
11. 验证部署状态，查看监控
```

## 场景二：日常迭代发布（金丝雀策略）

```
1. 开发者提交代码到 GitLab
2. Webhook 自动触发 CI 流水线构建
3. 流水线成功，产出新版本 Docker 镜像
4. 在流水线页面点击"部署上线" → 系统自动创建发布工单
5. 进入"发布管理"页面，点击"提交审批"
6. 审批人点击"审批通过"
7. 点击"执行发布" → 系统自动创建金丝雀工作负载（1副本）
8. 观察金丝雀实例运行状态和监控指标
9. 确认无异常后点击"确认全量" → 系统全量部署并清理金丝雀
10. 查看监控确认服务正常
```

## 场景三：线上问题紧急回滚

```
1. 在监控中心发现异常指标/告警
2. 进入部署管理，找到对应部署
3. 点击"回滚"操作，回退到上一稳定版本
4. 确认服务恢复正常
5. 排查问题后重新走发布流程
```
